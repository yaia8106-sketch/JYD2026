# DCache 配置分析报告

> 生成时间: 2026-04-28 (含时序分析更新)
> 工具: `02_Design/coe/cache_sim.py` (cycle-accurate, 匹配 RTL dcache.sv)
> 时序来源: `03_Timing_Analysis/reports/stage_timing_report.txt` (250MHz, worst slack = 0.082ns)

## 1. 仿真方法

- **ISA 模拟器**: RV32I 全指令模拟，从 COE 文件加载程序
- **Cache 模型**: Write-Through + Write-Allocate, LRU 替换, 1-entry Store Buffer
- **Cycle 模型** (匹配 RTL dcache.sv):
  - Refill penalty: `LINE_WORDS + DRAM_LATENCY + 1` (16B line = 9 cycles, 32B line = 13 cycles)
  - DRAM_LATENCY = 4 (registered addr + BRAM read + DOB_REG + dram_rdata_r)
  - SB conflict: 连续 store hit 时 SB 未排空 → 2-cycle stall
  - SB drain before refill: miss 时 SB pending → 额外 2 cycles
  - SB 后台排空: 非连续内存指令间 FSM 有时间排空 SB (基于 inst_idx 判断)
  - Load-use stall: N+1 = 2 cycles (load_in_ex), N+2 = 1 cycle (load_in_mem)
    - 仅对 cache hit 的 load 有效 (miss 的 refill stall 吸收 load-use stall)
- **测试程序**: current, src0, src1, src2 (各 5M 指令)
- **对比基准**: 无 DCache @ 200MHz (每次 load 付 DRAM_LATENCY=4 周期)
- **Speedup 公式**: `(base_CPI/200MHz) / (cache_CPI/250MHz)`

## 5. BRAM 使用分析

2-way cache 的 Data RAM 每个 way 占 1 个 BRAM18 (36b 模式, 512 entries):

| 配置 | entries/way | BRAM18 利用率 | Data BRAM18 | Tag RAM |
|------|:----------:|:------------:|:-----------:|:-------:|
| 2W 2KB/16B | 256 | 50% | 2 | LUTRAM |
| **2W 4KB/16B** | **512** | **100%** | **2** | **LUTRAM** |
| 2W 8KB/16B | 1024 | 100% | 4 | LUTRAM |

**关键**: 2W 2KB 和 2W 4KB 占用完全相同的 2 个 BRAM18，但 4KB 利用率 100%。

## 6. 时序分析 (250MHz)

> 数据来源: `03_Timing_Analysis/reports/stage_timing_report.txt`

### 6.1 DCache 相关关键路径

DCache 包含了**全局最差的时序路径**:

| # | 路径 | Slack (ns) | DataPath (ns) | Levels | 说明 |
|:-:|------|:----------:|:-------------:|:------:|------|
| 1 | DCache→DRAM | **0.082** | 3.482 | **0** | `dram_rd_addr` → DRAM BRAM 地址 (**全局最差, 纯布线延迟**) |
| 2 | DCache→IROM | 0.114 | 3.398 | 7 | `mem_index` → IROM BRAM 地址 (通过 pipeline stall) |
| 3 | ID/EX→DCache | 0.209 | 3.730 | 7 | EX 阶段地址 → DCache tag 读取 |
| 4 | DRAM→DCache | 0.290 | 3.354 | 3 | DRAM 读数据 → DCache refill 寄存器 |
| 5 | DCache→ID/EX | 0.304 | 3.325 | 7 | DCache stall → ID/EX 锁存使能 |
| 6 | DCache→DC_Data | 0.309 | 3.284 | 7 | DCache FSM → DC Data BRAM 控制 |
| 7 | DCache→MEM/WB | 0.421 | 3.488 | 2 | DCache/DC_Data 读数据 → MEM/WB |

### 6.2 2KB→4KB 时序影响分析: ✅ 安全

关键问题: DCache 已是全局瓶颈 (0.082ns slack), 扩容是否会恶化?

| 关键路径 | 是否受影响 | 原因 |
|----------|:----------:|------|
| **DCache→DRAM (0.082ns)** | **否** | `dram_rd_addr` 扇出到 64 个 DRAM BRAM 的**布线延迟**, 0 级逻辑。4KB 不改变 DRAM 地址位宽或扇出 |
| DCache→IROM (0.114ns) | 否 | 通过 pipeline stall 信号间接连接, cache 容量不影响 stall 逻辑深度 |
| ID/EX→DCache (0.209ns) | 微改善 | Cache index 从 5→6 bit, tag 从 N→N-1 bit, tag 比较逻辑反而简化 |
| DCache→DC_Data (0.309ns) | 否 | 每 way 仍是 1 个 BRAM18 (256→512 entries), 同一 BRAM 内部寻址不变 |
| DCache→MEM/WB (0.421ns) | 否 | DC_Data 读端口不变 (BRAM18 内部 512×36b 仍单周期读) |

**结论**: 全局最差路径 DCache→DRAM 是**纯布线问题** (64 个 DRAM BRAM36 的地址扇出), 与 cache 容量完全无关。2KB→4KB 仅改变 DC_Data BRAM18 的利用率 (50%→100%), 不增加任何逻辑层级或 BRAM 数量。**时序安全, 零风险。**

## 7. 方案推荐

### 推荐方案: 2W 4KB/16B

| 指标 | 当前 (2W 2KB) | 推荐 (2W 4KB) | 差异 |
|------|:------------:|:------------:|:----:|
| AvgHit% | 97.94% | 98.45% | +0.51% |
| AvgCPI@250 | 1.167 | 1.162 | -0.005 |
| AvgSpeedup | +44.0% | +44.6% | +0.6% |
| BRAM18 | 2 | 2 | **0** |
| src0 Hit% | 95.78% | 97.83% | **+2.05%** |
| src0 CPI | 1.274 | 1.255 | **-0.019** |
| 时序风险 | — | ✅ 零 | 关键路径不变 |

**理由**:
1. **零 BRAM 成本**: 每 way 的 BRAM18 从 50% 利用率提升到 100%, 不增加 BRAM 数量
2. **零时序风险**: DCache→DRAM 全局最差路径是纯布线延迟 (64 个 DRAM BRAM 扇出), 与 cache 容量无关
3. **src0 显著改善**: 命中率 +2.05%, refill stall 减少 127K cycles
4. **无负面影响**: 其他程序工作集 < 2KB, 不受影响
5. **RTL 改动极小**: 仅需修改 SETS 参数 (64→128), 增加 1 bit index

### 不推荐的方案

| 方案 | 原因 |
|------|------|
| 2W 8KB | +0.05% hit vs 4KB, 但 BRAM 翻倍 (2→4), 收益饱和 |
| 4W 2KB | 比 2W 4KB 差 (hit 97.53% vs 97.83%), BRAM 更多 (4 vs 2) |
| 32B line | 在 2KB 时反而更差; 在 4KB 时虽好但 refill 13 cycles (vs 9), RTL 改动大 |
| DM 任何 | 缺乏抗冲突能力, 同容量下 hit 率不如 2-way |

## 8. 其他瓶颈 (cache 无法优化)

1. **SB conflict (src1)**: 97K 连续 store 对 → 130K stall cycles, 需要 2-entry SB 或 write-back 才能改善
2. **Load-use stall**: 各程序 226K-1.16M raw stall cycles, 由指令调度决定, 与 cache 配置无关
3. **分支预测**: 独立于 cache, 见 `bp_analysis.md`
4. **DRAM 布线瓶颈**: 全局 worst slack 0.082ns (DCache→64×DRAM BRAM), 需优化 DRAM IP 放置或约束
