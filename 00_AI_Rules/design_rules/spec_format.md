# Module Spec 编写规范

> 本文档定义 `02_Design/spec/` 下所有模块规格文件的格式要求。
> AI 生成 RTL 时，以 `pipeline.md`（全局协议）+ 对应 `_spec.md`（局部行为）为唯一依据。

---

## 1. 设计原则

**自包含性**：每份 spec 必须包含 AI 实现该模块所需的全部信息。AI 不需要阅读其他模块的代码或 spec。

**无歧义性**：所有行为必须明确描述，不留"常识"推断空间。AI 不了解你的意图，只理解你写的文字。

**可引用性**：如果某个行为依赖全局规则（如握手协议、BRAM 时序），用「见 `pipeline.md` 第 X 节」引用，不要复制粘贴。

---

## 2. 必需章节

每个 `_spec.md` **必须**包含以下章节：

### 2.1 端口列表

列出每个信号的名称、方向、位宽、含义。使用 Markdown 表格格式：

```markdown
| 信号名 | 方向 | 位宽 | 含义 |
|--------|------|------|------|
| clk | input | 1 | 系统时钟 |
| rst_n | input | 1 | 异步低有效复位 |
| ...    | ...  | ... | ... |
```

**要求**：
- 位宽用数字表示（如 `32`），参数化时写明参数名和默认值（如 `DATA_WIDTH (32)`）
- 信号命名遵循全局命名约定（见 `pipeline.md` 第 1.2 节）
- 明确标注哪些信号是组合逻辑输出、哪些是寄存器输出

### 2.2 功能描述

描述模块的核心行为。区分：
- **组合逻辑模块**：用真值表或 `case` 伪代码描述输入→输出映射
- **时序逻辑模块**：描述每个 posedge 的行为，用时序图说明多周期行为

### 2.3 时序约束

- 本模块的输出在哪个时钟沿稳定？
- 本模块的输入在哪个时钟沿被采样？
- 组合逻辑延迟的预估（是否在关键路径上）

### 2.4 边界条件

必须明确以下行为：
- **复位（`rst_n = 0`）**：所有寄存器的初始值
- **Flush**：收到 flush 信号时的行为（适用于级间寄存器）
- **Stall（`allowin = 0`）**：被阻塞时的行为

### 2.5 依赖文档

列出 AI 实现该模块时需要**同时阅读**的其他文档：

```markdown
- `pipeline.md` 第 2.2 节（IROM 时序）
- `design_rules/isa_encoding.md`（指令编码表）
```

---

## 3. 可选章节

根据模块类型，可以增加：

- **前递/旁路逻辑**：前递 MUX 的选择条件和优先级
- **参数化**：可配置的参数及其合法取值范围
- **综合约束**：是否需要映射到特定硬件资源（如 BRAM、DSP）
- **验证要点**：该模块容易出错的场景，供集成测试参考

---

## 4. 反面示例——不要做的事

| ❌ 不要 | ✅ 应该 |
|---------|---------|
| "ALU 支持常规运算" | 列出每个 alu_op 值对应的运算（0001=ADD, 0010=SUB, ...） |
| "flush 时清除数据" | "flush 时 `xx_valid <= 0`，数据域不关心（don't care）" |
| "参考其他模块的实现" | 把依赖的信号和行为在本 spec 中显式描述 |
| 复制粘贴 pipeline.md 的内容 | 引用："握手协议见 `pipeline.md` 第 4 节" |
