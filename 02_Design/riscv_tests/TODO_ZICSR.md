# Zicsr 与 Trap 测试接入 TODO

> 本文只描述改动逻辑、测试范围和验收路径，不记录具体实现语句。

## 背景与目标

分赛区决赛会测试 Zicsr 指令集和最小 Trap 机制。测试范围集中在 M 模式下的基础能力：

- 支持 Zicsr 的六类读改写行为。
- 支持 `mstatus`、`mepc`、`mtvec`、`mcause` 四类基础 CSR。
- `mstatus` 重点覆盖 `MIE` 和 `MPIE` 两个位。
- `mtvec` 只要求 Direct 模式，并指向异常处理入口。
- 至少支持 `ecall` 触发 Trap，并支持 `mret` 从异常处理流程返回。
- CPU 执行测试程序时按 M 模式处理，不扩展到多特权级切换。

总体目标是：在不破坏现有 RV32I 和双发射回归的基础上，让处理器能够通过区域赛给出的 Zicsr、异常处理和快速验证程序。

## 阶段 0：需求冻结与基线确认

- 明确首轮只覆盖赛题文档列出的 M 模式最小集合。
- 暂不把中断、Vectored Trap、多特权级、完整 `mstatus` 字段、性能计数 CSR 作为首轮目标。
- 记录当前 RV32I 与双发射测试通过状态，作为后续改动的回归基线。
- 确认测试程序的结束判定、异常处理流程和显示结果与现有验证方式兼容。

## 阶段 1：CSR 行为模型

- 建立最小 CSR 状态集合：
  - `mstatus`：只保证 `MIE`、`MPIE` 的可读写和 Trap 期间更新。
  - `mepc`：保存异常触发指令地址，也允许异常处理流程读写。
  - `mtvec`：保存 Direct 模式异常入口地址。
  - `mcause`：保存异常原因，首轮至少覆盖 M 模式下的 `ecall`。
- CSR 指令行为按“先读旧值，再按指令语义更新 CSR，再把旧值送回目标寄存器”的原则处理。
- 遵守零寄存器相关规则：
  - 目标寄存器为零时，不需要产生普通寄存器写回结果。
  - 源寄存器或立即数字段为零时，对应的置位/清位类 CSR 操作不改变 CSR。
- 对未纳入首轮范围的 CSR，采用明确且一致的策略，避免测试中出现不可预测状态。

## 阶段 2：Trap 与返回逻辑

- `ecall` 被识别为同步异常，异常入口由 `mtvec` Direct 模式给出。
- 触发 Trap 时：
  - `mepc` 保存触发 `ecall` 的指令地址。
  - `mcause` 保存 M 模式环境调用原因。
  - `mstatus.MPIE` 保存 Trap 前的 `MIE`。
  - `mstatus.MIE` 在进入 Trap 后清零。
  - 后续取指跳转到 `mtvec` 指定的异常处理入口。
- 异常处理流程可以修改 `mepc`，例如跳过触发异常的指令。
- `mret` 返回时：
  - 取指跳转到当前 `mepc`。
  - `mstatus.MIE` 从 `MPIE` 恢复。
  - `mstatus.MPIE` 恢复为 1。
- 首轮只处理精确同步异常，不引入异步中断。

## 阶段 3：流水线与双发射约束

- CSR 操作、`ecall`、`mret` 按顺序可见原则处理，避免错误路径修改 CSR。
- Trap 和返回属于控制流重定向事件，必须清理更年轻的错误路径指令。
- CSR 相关指令首轮按串行化类别处理，避免同包双发射造成 CSR 读写顺序不清。
- CSR 读结果需要能被紧随其后的普通指令正确使用。
- Trap 发生时，不能让同周期或后续错误路径的访存、寄存器写回、CSR 修改泄漏出去。

## 阶段 3.5：微架构决策

- CSR、`ecall`、`mret` 都归类为系统类顺序化指令。
- 系统类指令只允许在 Slot 0 执行。
- Slot 0 为系统类指令时，Slot 1 不发射。
- 系统类指令出现在 Slot 1 位置时，不与 Slot 0 同包执行，而是在后续周期作为 Slot 0 执行。
- CSR 状态修改、Trap 状态更新、`mret` 状态恢复必须等到指令确认有效后再生效，不能让错误路径指令修改 CSR。
- CSR 读出的旧值作为普通寄存器写回结果处理，后续普通指令通过既有数据依赖处理机制获得该结果。
- 背靠背 CSR 按严格顺序执行，后一条 CSR 必须观察到前一条 CSR 的最终状态。
- `ecall` 保存自身指令地址到 `mepc`，进入 `mtvec` Direct 入口，并清理所有更年轻的错误路径指令。
- `mret` 使用当前 `mepc` 作为返回目标，并按 M 模式返回规则恢复 `mstatus.MIE` 和 `mstatus.MPIE`。
- 如果更老的控制流重定向与较新的系统类指令同时出现，更老事件优先；较新的系统类指令不得产生可见副作用。
- 系统类重定向复用统一的前端清理语义，避免异常返回和普通分支跳转形成两套不一致规则。
- `ecall` 和 `mret` 不作为普通分支或跳转参与分支预测状态更新。

## CSR 指令语义细节

- CSR 指令统一遵循“读旧值、计算新值、旧值写回目标寄存器”的顺序语义。
- 目标寄存器为零时，旧值不写入通用寄存器，但 CSR 是否修改仍由具体 CSR 指令类型决定。
- 写交换类 CSR 指令即使源值为零，也会把零写入目标 CSR。
- 置位类和清位类 CSR 指令在源寄存器为零时只读不改。
- 立即数字段版本的置位类和清位类 CSR 指令在立即数字段为零时只读不改。
- 立即数字段版本的写交换类 CSR 指令即使立即数字段为零，也会把零写入目标 CSR。
- 未纳入首轮范围的 CSR 首轮采用一致的宽松处理策略：读取得到零，写入不改变状态，不主动引入非法指令异常。
- `mtvec` 首轮只使用 Direct 模式入口。读回时保留写入值；取异常入口时按 Direct 基址处理。

## 阶段 4：测试内容设计

- Zicsr 基础测试：
  - 覆盖读写、置位、清位，以及立即数字段版本。
  - 覆盖旧值返回是否正确。
  - 覆盖零寄存器和零立即数的无修改语义。
  - 覆盖 CSR 结果被下一条指令立即使用。
- CSR 寄存器测试：
  - `mstatus.MIE` 与 `mstatus.MPIE` 的读写组合。
  - `mtvec` 写入异常处理入口后读回。
  - `mepc` 普通读写，以及 Trap 后自动保存。
  - `mcause` 在 `ecall` 后为预期异常原因。
- Trap 流程测试：
  - 设置异常入口，触发 `ecall`，进入异常处理流程。
  - 异常处理流程检查 `mepc`、`mcause`、`mstatus`。
  - 异常处理流程调整返回地址后执行 `mret`。
  - 返回主流程后继续执行并给出通过结果。
- 交互测试：
  - CSR 指令前后穿插普通算术、访存和分支。
  - Trap 前后检查错误路径指令不会提交。
  - 双发射窗口中遇到 CSR 或 Trap 类指令时，验证顺序语义稳定。

## 阶段 5：验证与文档同步

- 先跑现有 RV32I 与双发射回归，确认基线不退化。
- 再单独运行 Zicsr 基础测试，定位 CSR 语义问题。
- 再运行 Trap 流程测试，定位异常入口、返回地址和状态位问题。
- 最后运行包含普通指令、CSR、Trap 混合场景的综合测试。
- 通过后同步更新测试覆盖说明和架构说明，明确当前支持的是 M 模式最小 Zicsr/Trap 子集。

## 首轮验收标准

- 现有 RV32I 与双发射测试全部保持通过。
- Zicsr 六类基础行为全部通过。
- `mstatus.MIE`、`mstatus.MPIE`、`mepc`、`mtvec`、`mcause` 的读写和 Trap 更新符合预期。
- `ecall` 能进入异常处理流程，`mret` 能返回预期地址。
- 错误路径不会产生寄存器、访存或 CSR 可见副作用。

## 主要风险点

- `mepc` 保存的是触发异常的指令地址，不是下一条指令地址。
- Trap 入口必须使用 `mtvec` Direct 模式目标。
- `mstatus.MIE` 和 `MPIE` 在进入 Trap 与执行 `mret` 时的交换规则容易写反。
- CSR 修改必须按有效提交顺序生效，否则分支错误路径可能污染状态。
- 双发射环境下需要避免 CSR 与 Trap 类指令产生同包顺序歧义。
- CSR 指令的零寄存器和零立即数字段语义容易混淆，尤其写交换类与置位/清位类不同。
- `ecall`、`mret` 不应污染分支预测状态，也不应被普通跳转预测机制误处理。
- 更老访存未完成时，系统类指令不能提前产生异常入口或返回重定向。

## 已确认边界

- `mtvec` 读回保留写入值，异常入口只按 Direct 基址使用。
- 未支持 CSR 采用“读零、写忽略”的宽松策略，不在首轮补非法指令异常。
- 首轮不补 `mcycle`、`minstret` 等计数类 CSR。
